Spark源码调试（第一步）

环境

1.Java 1.8

2.Scala

3.Maven

4.Git

5.Intellij Idea

源码下载

下载spark源码,http://spark.apache.org/downloads.html 选择source code

编译

安装Scala插件

File–>Settings–>Plugins–>Install JetBrains plugin

配置Maven

在编译之前，首先需要对Maven的相关参数进行配置，以确保编译过程中不会出错，在Maven安装目录下找到bin/mvn.cmd文件，打开在文件中添加

MAVEN_OPTS=”-Xmx2g -XX:ReservedCodeCacheSize=512m”

VM options: -Dspark.master=local 代表使用本地模式运行Spark代码，也可以选择其他模式。

问题

运行后报错 Exception in thread “main” java.lang.NoClassDefFoundError: scala/collection/Seq

解决方法：

Menu -> File -> Project Structure -> Modules -> spark-examples_2.11 -> Dependencies 添加依赖 jars -> {spark dir}/spark/assembly/target/scala-2.11/jars/

重新运行

成功

MRv1存在以下不足：

可拓展性差：运行时，JobTracker既负责资源管理又负责任务调度

可用性差：单节点

资源利用率低：TaskTracker使用slot灯亮互粉本届电商的资源量，solt代表计算资源，一个Task获取到一个slot后才有机会运行，Hadoop调度器负责将各个TaskTracker上的空闲Slot分配给Task使用。一些Task并不能充分利用slot，其他Task也无法使用这些空闲资源。

不能支持多种MR框架：无法通过可查吧的方式将自身的MR框架替代为其他实现，这时ReduceSlot也会被闲置。

MRv1为了解决以上问题，对Hadoop进行了改造，诞生了MRv2，重用了MRv1中的编程模型和数据处理引擎，但是运行时环境被重构了。JobTracker被拆成了通用的资源调度平台（RM）和负责各个计算框架的任务调度模型，MRv2的核心不再是MR框架，而是Yarn，在以Yarn为核心的MRv2种，MR的框架是可插拔的，完全可以替代为其他MR实现，比如Spark Storm，Hadoop MRv2虽然解决了MRv1的一些问题，，但是对于HDFS的频繁操作，包括结果持久化，数据备份 shuffle等，导致IO成为系统性能的瓶颈。
